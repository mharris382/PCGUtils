Implement a new **PCGUtilsDataCache** module inside the PCGUtils plugin.

## Feature overview

Create an alternative cross-actor PCG communication channel that uses generated `UPCGDataAsset` assets instead of relying on a `UPCGComponent`’s generated output cache.

The normal workflow:

```text
Get Actor Data
    -> PCG Component Output
```

makes it easy for one actor to compute high-level PCG data and for several other actors to reuse that data without recomputing it.

However, PCG component output data is serialized with the level and has several disadvantages:

1. Large, dense, high-resolution datasets can substantially increase level size and memory use.
2. Offline generation data and runtime generation data must share the same PCG component output cache.
3. Offline-only data remains associated with and loaded alongside the level.
4. Serialized PCG output may be cooked even when it is never required at runtime.

The new workflow should be:

```text
Save PCG Cache Data
    -> generated UPCGDataAsset

Get PCG Cache Data
    -> soft object path attribute
    -> vanilla Load PCG Data Asset node
```

The system should support two logically separate caches:

```cpp
UENUM(BlueprintType)
enum class EPCGUtilsDataCache : uint8
{
    Offline,
    Runtime
};
```

* **Offline cache:** intended for editor-only or offline generation data.
* **Runtime cache:** intended for data that will be loaded in packaged builds.

Both caches use the same mechanism but save to different configurable content folders.

Important distinction:

* “Runtime cache” means that the generated data asset may be cooked and loaded at runtime.
* Saving or modifying `UPCGDataAsset` packages remains an editor-only operation.
* Packaged builds must never attempt to create or save asset packages.

# Initial investigation

Before implementing:

1. Inspect the existing PCGUtils plugin structure, naming conventions, module setup, logging conventions, and current supported Unreal Engine version.
2. Inspect the Unreal Engine 5.8 implementations of:

   * `UPCGDataFromActorSettings`
   * `FPCGDataFromActorElement`
   * `FPCGDataFromActorElement::ProcessActor`
   * `UPCGSaveDataAssetSettings`
   * its corresponding execution element
   * `UPCGDataCollectionExporter`
   * `UPCGLoadDataAssetSettings`
3. Reuse current engine patterns where appropriate instead of reimplementing asset export behavior from scratch.
4. Do not subclass `UPCGSaveDataAssetSettings`, because the cache node has different settings and must not expose arbitrary asset paths, save names, or save dialogs.
5. The new Get node should derive from the actor-data query infrastructure so it retains standard actor selection and tracking behavior.

Use the APIs actually present in this engine branch rather than assuming older PCG APIs.

Create a dedicated log category for this module.

# Shared cache path conventions

Create a shared helper or namespace for cache path construction so the component, Get node, and Save node cannot independently construct different paths.

Define a shared PCG metadata attribute name:

```cpp
CacheAssetPath
```

This attribute should contain an `FSoftObjectPath` pointing to the corresponding `UPCGDataAsset`.

Use a deterministic asset name derived from the owning actor’s persistent Actor GUID.

A suitable format is:

```text
PCGCache_<ActorGuidDigits>
```

Do not include the actor label or actor object name in the authoritative asset name because actor renaming must not change the cache path.

The full package path should be:

```text
<ConfiguredCacheFolder>/<GeneratedSaveName>
```

and the object path should point to the object with the same name inside that package.

Centralize:

* folder normalization;
* package-name validation;
* asset-name generation;
* object-path generation;
* target-cache availability checks.

Do not store either cache asset as a serialized `TSoftObjectPtr<UPCGDataAsset>` on the component. In particular, the offline cache must not create a serialized soft reference that could cause it to be included in a cook.

# PCGDataCacheComponent

Create:

```cpp
UPCGDataCacheComponent : public UActorComponent
```

The component identifies an actor as a source of generated PCG cache data.

Use an appropriate component class group and make it Blueprint-spawnable.

It does not need to tick.

## Properties

Expose:

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache")
bool bUsesRuntimeCache = true;

UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache")
bool bUsesOfflineCache = true;
```

Also expose the metadata copied onto generated `UPCGDataAsset` assets:

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache|Asset")
FText CacheDescription;

UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache|Asset")
FLinearColor CacheColor;
```

Use a sensible default color.

## Persistent cache name

The component must own a serialized, globally unique cache save name derived from the owning actor’s Actor GUID.

For example:

```cpp
UPROPERTY(VisibleAnywhere, Category="Data Cache", AdvancedDisplay)
FName CacheSaveName;
```

The persistent name is important because the actor’s editor Actor GUID may not be available through the same API in packaged builds.

Requirements:

1. In editor builds, synchronize `CacheSaveName` from the owner’s persistent Actor GUID.
2. Do not create a new random GUID as a fallback on every query.
3. Do not base the name on actor label or actor object name.
4. When an actor is duplicated, the duplicated actor should receive a path based on its new Actor GUID.
5. Handle the appropriate editor lifecycle callbacks so duplication, loading, and saving leave the component synchronized.
6. Do not regenerate or mutate the name in packaged builds.
7. If the owner does not have a valid persistent Actor GUID when synchronization is required, report a clear error rather than silently producing an unstable path.

Inspect the current `AActor` lifecycle and GUID APIs and choose the safest callbacks. Do not assume the GUID is valid during the component constructor.

## Blueprint API

Expose a generic function:

```cpp
UFUNCTION(BlueprintPure, Category="PCG Utils|Data Cache")
bool GetCacheAssetPath(
    EPCGUtilsDataCache TargetCache,
    FSoftObjectPath& OutAssetPath) const;
```

Optional convenience wrappers may also be provided:

```cpp
GetRuntimeCacheAssetPath
GetOfflineCacheAssetPath
```

Also expose a query such as:

```cpp
bool UsesCache(EPCGUtilsDataCache TargetCache) const;
```

Behavior:

* Return `false` and an invalid path if the selected cache is disabled.
* Return `false` if project settings or the generated save name are invalid.
* Runtime cache paths may be resolved in editor and packaged builds.
* Every exposed offline-cache path query must return `false` and an invalid path in non-editor packaged builds.
* Path queries must not load the asset.

Assume one `UPCGDataCacheComponent` per actor. Add clear validation or warnings if multiple cache components are found on the same actor rather than silently producing duplicate references.

# Get PCG Cache Data node

Create a PCG node named:

```text
Get PCG Cache Data
```

Suggested classes:

```cpp
UPCGGetPCGCacheDataSettings
FPCGGetPCGCacheDataElement
```

The settings should derive from:

```cpp
UPCGDataFromActorSettings
```

The element should derive from:

```cpp
FPCGDataFromActorElement
```

and specialize actor processing through the appropriate `ProcessActor` or `ProcessActors` override.

Retain the normal actor selector and actor dependency tracking behavior from the base implementation, but hide or disable inherited data-retrieval settings that are irrelevant to this node. The user should not be asked to choose normal modes such as parsing components or retrieving PCG component output.

## Settings

Expose:

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache")
EPCGUtilsDataCache TargetCache = EPCGUtilsDataCache::Runtime;
```

Add an output representation setting:

```cpp
UENUM(BlueprintType)
enum class EPCGUtilsDataCacheReferenceOutput : uint8
{
    PointData,
    AttributeSet
};
```

Expose it as:

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache")
EPCGUtilsDataCacheReferenceOutput OutputType =
    EPCGUtilsDataCacheReferenceOutput::PointData;
```

Add an advanced warning suppression option:

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache", AdvancedDisplay)
bool bSuppressSelfCacheWarning = false;
```

Use current preferred PCG point data types. Prefer `UPCGPointArrayData` over legacy `UPCGPointData` if that is the correct API for this engine version and is consistent with the rest of PCGUtils.

## Processing each actor

For each actor selected by the inherited actor selector:

1. Look for a `UPCGDataCacheComponent`.
2. If none exists, skip the actor. Broad actor queries should not emit a warning for every actor without the component.
3. If multiple components exist, issue one clear warning for that actor and use a deterministic policy. Prefer treating this as invalid and skipping the actor.
4. Check whether the component allows the selected `TargetCache`.
5. Ask the component for the corresponding soft object path.
6. If the path is invalid, issue a useful warning and skip the actor.
7. Do not load the `UPCGDataAsset`.
8. Output the soft object path in the shared `CacheAssetPath` attribute.

### PointData output

Output one point for each valid source actor.

The point transform must use the source actor’s world transform.

The point metadata must contain:

```text
CacheAssetPath : FSoftObjectPath
```

Use sensible default point density, bounds, and seed values consistent with a single actor-reference point.

### AttributeSet output

Output one metadata entry for each valid source actor containing:

```text
CacheAssetPath : FSoftObjectPath
```

Prefer preserving the distinction between source actors rather than accidentally overwriting entries.

The output pin’s advertised type should reflect the selected representation where supported by the PCG node API.

## Self-selection warning

If the inherited actor selector is configured to query `Self`:

* Allow the query normally.
* If `bSuppressSelfCacheWarning` is false, inspect the source actor’s PCG components.
* If the actor has zero or one PCG component, issue one warning explaining that loading cache data saved by the same actor may be confusing or indicate an accidental self-reference.
* If the actor has more than one PCG component, do not issue this warning because one component may legitimately consume cache data generated by another component.
* This is only a warning and must never block execution.

Suggested warning meaning:

```text
Get PCG Cache Data is querying Self. Loading cache data saved by the same actor may create a confusing or circular workflow. This is valid when separate PCG components on the actor produce and consume the cache.
```

Avoid logging the warning once per selected actor or once per point.

## Runtime behavior

* Runtime cache references must work in packaged builds.
* Offline cache queries must produce no output in packaged non-editor builds.
* Offline queries in packaged builds may log one non-spammy warning explaining that offline caches are editor-only.
* The Get node must not synchronously or asynchronously load the asset itself.

The output is specifically intended to feed the vanilla `Load PCG Data Asset` node with `Load From Input` enabled and its asset-reference selector configured to read `CacheAssetPath`.

# Save PCG Cache Data node

Create a PCG node named:

```text
Save PCG Cache Data
```

Suggested classes:

```cpp
UPCGSavePCGCacheDataSettings
FPCGSavePCGCacheDataElement
```

The settings must derive directly from `UPCGSettings`, not from `UPCGSaveDataAssetSettings`.

## Settings

Expose only:

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Data Cache")
EPCGUtilsDataCache SaveTargetCache =
    EPCGUtilsDataCache::Offline;
```

Do not expose:

* arbitrary package path;
* arbitrary asset name;
* a save dialog option;
* overwrite prompts;
* exporter paths that allow bypassing the component;
* any setting that lets the node save outside the configured cache folders.

The default may be Offline because large editor-generation datasets are the primary motivation for this feature.

## Pins

The node should accept the PCG data collection to save.

Follow the vanilla Save PCG Data Asset node’s pin and pass-through behavior as closely as practical:

* Preserve the input data collection.
* Save the complete collection, including pin labels and tags, using the standard PCG data-asset export representation.
* Pass the original input through the output if the vanilla node does so.
* A failed or unavailable save should not unexpectedly destroy downstream data.

Inspect the current vanilla node implementation rather than relying on memory.

## Execution

The save operation is editor-only.

During execution:

1. Resolve the source `UPCGComponent` and its owning actor from the PCG context.
2. Find the actor’s `UPCGDataCacheComponent`.
3. If no component exists, log a clear PCG graph error and do not save.
4. If multiple components exist, log an error and do not save.
5. Verify that the selected `SaveTargetCache` is enabled by the component.
6. Ask the component for the deterministic cache path.
7. Validate the package path and object name.
8. Create a new `UPCGDataAsset` or update the existing asset at exactly that path.
9. Export the complete input `FPCGDataCollection` into the asset using the same current engine mechanisms as the vanilla Save PCG Data Asset node.
10. Copy the component’s:

    * `CacheDescription` to the asset description;
    * `CacheColor` to the asset color.
11. Mark the package dirty and save it using the appropriate editor APIs.
12. Never open a save dialog.
13. Never generate a unique suffix because the asset already exists.
14. Regenerating the actor should deterministically update the same asset.
15. Emit a useful success or failure message through the PCG log.

Where practical, reuse engine exporter helpers. A carefully adapted copy of the vanilla element’s execution logic is acceptable, but remove all save-dialog and user-selected-path behavior.

## Packaged builds

In non-editor builds:

* The node must compile.
* It must never attempt to create packages, modify assets, access editor subsystems, or write files.
* It should act as a no-op/pass-through node.
* It may emit one non-spammy warning explaining that PCG data cache assets must be generated in the editor.
* This applies to both Runtime and Offline targets.

Again, the Runtime target means “available for loading at runtime,” not “saved as a `.uasset` during runtime gameplay.”

---

# Asset update and stale data behavior

When an asset already exists:

* Replace or update its stored PCG data collection using the current vanilla exporter behavior.
* Update its description and color.
* Preserve the deterministic package and object name.
* Do not create redirectors or numbered duplicate assets.
* Do not prompt the user.
* Ensure editor asset registry state is updated correctly.

This initial implementation does not need automatic stale-cache deletion.

Do not automatically delete an old cache asset if:

* an actor is deleted;
* its component is removed;
* its Actor GUID changes;
* the configured cache folder changes.

That can be handled by a future cache-management utility.

# Validation and diagnostics

Provide useful diagnostics for:

* missing cache component on the actor executing the Save node;
* multiple cache components;
* target cache disabled;
* invalid project cache folder;
* invalid or unsynchronized cache save name;
* failure to create or find the package;
* failure to export the PCG data collection;
* failure to save the package;
* attempting to use offline cache data in a packaged build;
* attempting to execute the Save node outside the editor;
* suspicious Self queries in the Get node.

Avoid noisy warnings when a broad Get-node actor selection simply includes actors that do not have a cache component.

Use PCG graph logging where possible so errors appear on the relevant node.

---

# Editor and cooking constraints

The implementation must preserve these guarantees:

1. The runtime module compiles without editor functionality in packaged builds.
2. Offline cache paths are inaccessible through the public component API in packaged builds.
3. No component property stores a soft reference to an offline cache asset.
4. The Get node only constructs and outputs a path; it does not load assets.
5. Runtime cache assets can be cooked when referenced or included by the project.
6. Offline cache assets are eligible to remain uncooked as long as the project does not explicitly cook their directory or all project content.
7. Editor-only saving code is protected by build configuration and module dependency guards.

# Acceptance criteria

The implementation is complete when all of the following work:

## Deterministic paths

* Adding a cache component to an editor-placed actor produces a save name based on that actor’s persistent Actor GUID.
* Renaming the actor does not change its cache path.
* Duplicating the actor results in a different cache path.
* Reloading the level does not change either path.

## Saving

* A graph running on an actor with a cache component can save Offline cache data.
* The same graph can save Runtime cache data.
* The paths use the corresponding project-settings folders.
* Regeneration updates the same asset without a dialog.
* The complete PCG data collection is preserved.
* Asset description and color come from the component.
* Disabled cache targets do not save.

## Retrieving

* Another actor can use Get PCG Cache Data to find the source actor.
* Point output creates one point at the source actor transform.
* AttributeSet output creates one entry for the source actor.
* Both outputs contain a valid `CacheAssetPath` soft object path.
* The Get node does not load the asset.
* The output can drive the vanilla Load PCG Data Asset node in load-from-input mode.

## Warning behavior

* A Self query on an actor with one PCG component raises the explanatory warning.
* The same query still produces output.
* The warning can be suppressed.
* An actor with multiple PCG components does not raise the Self warning.
* Broad actor queries do not spam warnings for actors without cache components.

## Runtime/editor separation

* Editor builds can generate both cache asset types.
* Packaged builds can retrieve Runtime cache paths.
* Packaged builds cannot retrieve Offline cache paths.
* Save PCG Cache Data is a safe pass-through/no-op in packaged builds.
* Packaged builds contain no unconditional editor module dependencies.
* The plugin compiles in Editor, Development, and Shipping targets.

# Scope constraints

Keep this implementation focused.

Do not add:

* automatic cache cleanup;
* a cache browser;
* runtime `.uasset` creation;
* user-selected filenames;
* save dialogs;
* automatic packaging-settings modification;
* automatic loading inside Get PCG Cache Data;
* an additional custom Load node;
* support for multiple independent cache components on one actor;
* cache versioning or historical snapshots;
* network replication.

Use clean, documented C++ and follow PCGUtils conventions.

After implementation, report:

1. all added or modified files;
2. the class and module architecture;
3. how the Actor GUID is synchronized into the persistent save name;
4. how editor-only saving is isolated from packaged builds;
5. how the Get node’s output connects to Load PCG Data Asset;
6. build or test results;
7. any engine API limitations or remaining caveats.
